版本控制 - Git原理

Posted on 2019-07-15

版本控制是用来管理项目版本的，它有以下几个好处：

  1. 记录每一次提交，方便之后恢复到以前的任一版本
  2. 多人协同编写代码，独立分支开发，合并多个分支的代码
  3. 出现问题时，记录的提交记录方便定位问题
  4. 代码、资料等文件备份

版本控制区分

本地版本控制

利用简单的版本数据库来记录文件的历次更新差异

集中式版本控制（CVCS，Centralized Version Control Systems）

由单一集中服务器来管理，保存所有文件的修订版本。所有工作的客户端连接到这台服务器，取出文件或者提交更新

分布式版本控制（DVCS，Distributed Version Control System）

客户端完整拷贝代码仓库，保存所有文件的修订版本。每一次的提取操作，实际上都是对代码仓库的完整备份

集中式 & 分布式

版本控制类型 代表工具 优势 劣势
集中式版本控制 SVN 1. 理解成本低，操作简单
2. 更方便配置文件目录权限		 1. 客户端未连接到集中服务器时无法提交修改
2. 单点故障时无法协同工作
3. 如果中央服务器磁盘发生故障，则会发生数据丢失的风险，如丢失历史提交记录
4. 服务器压力大，数据库容量暴增
分布式版本控制 Git 1. 提交时无需联网
2. 任一协同服务器发生故障，事后都可以用任何一个本地仓库恢复
3. 代码回退和分支管理等操作效率更高		 1.学习理解成本高
2.难以按目录层级设置权限

Git原理

Git的三大区域

  • 工作区（Working Directiory）: 本地文件代码所在目录
  • 本地仓库（Repository）: .git目录，也叫做版本库。用于进行版本控制
  • 暂存区（Staging Area）: .git/index文件。代码commit之前需要提交到暂存区

Git常用命令

  • add: 将已修改的文件添加到缓存区
  • commit: 将缓存区中记录的文件同步到master分支。master分支增加新的commit记录，并将master指针指向最新记录
  • checkout: 可用于创建切换分支、将暂存区内容覆盖到工作区、将已提交的文件覆盖工作区和暂存区
  • rm: 删除暂存区和工作区的文件。加上–cache参数则只从版本控制中移除
  • reset: 操作版本提交历史记录，修改head指针到制定版本
  • status: 查看暂存区的内容

为了便于理解文件快照，可以参考下面这段说明：

我们都知道当你删除一个文件后，文件原来所占的磁盘空间并不是被清空，而是被文件系统标记为“已废弃，可修改”的状态，快照的作用就相当于将旧文件所占的空间保留下来，并且保存一个引用，而新文件中会继续使用与旧文件内容相同部分的磁盘空间，不同部分则写入新的磁盘空间。总的来说git其实也算是保存diff的方式，只不过是在文件系统层次上实现的。

摘自：https://www.zhihu.com/question/27680108/answer/156748005

文件快照可以理解为一份文件的完整内容，只不过Git做了存储上的优化。所以Git在版本控制时并不需要比较文件差异，例如在回退版本时直接按快照内容读取目录树和文件内容即可

版本管理 - Git Objects

Git使用对象来存储版本信息，这些对象都以文件形式保存在.git/obejcts目录下。Git对象分为以下几种：

  • blob: 数据对象。用于记录文件快照内容
  • tree: 树对象。对应工作区的文件目录。tree对象可以包含一个或多个tree记录，每条记录可以包含一个指向blob或tree对象的指针
  • commit: 提交对象。用于保存所有快照信息，也记录谁、何时保存了这些快照。 除了首次提交外，提交对象还会包含一个指向上次提交对象的指针

关于文件索引

文件索引是使用SHA-1算法（安全哈希算法，创建160位的哈希值，用16进制表示则是40位）来计算的哈希值。Git中所有数据都要进行内容的哈希值计算，并以这个结果作为对象的索引

每当使用git commit新建一个提交对象之前，Git会先计算每一个子目录及其所有内容的哈希值。

分支管理

分支: 实际上就是一个文件，它指向提交对象的哈希值

HEAD指针: Git中用来指向当前正在工作的本地分支的指针，即记录当前工作分支

以下通过一个简单的例子来理解分支创建合并的过程（红色箭头为HEAD指针）：

1) 当前正在master分支上工作，并有三次提交记录C0、C1、C2

2) 创建并切换到iss53分支

3) iss53编辑文件并提交更新，iss53指针指向最新提交的C3提交对象的索引

4) 有紧急问题需要修补，回到master分支并新建hotfix分支，hotfix指针指向最新提交的C4提交对象的索引

5) 问题已经修复，回到master分支且合并hotfix分支的修改内容

这种单线历史分支的合并并不需要解决任何分歧，因此这种分支合并过程被称为快进(Fast forward)。这种合并模式不会生成新的commit记录，如果需要生成新的commit记录可以加上—no-ff参数

6) 删除hotfix分支

回到iss53分支上继续提交更新，iss53指针指向最新提交的C5提交对象的索引

7) 回到master分支，然后合并iss53的修改内容

合并前master的提交对象C4并不是iss53的提交对象C5的直接祖先，因此Git会用两个分支的末端C4和C5，再加上两个分支的共同祖先C2进行一次合并计算

合并后生成一份新的快照和提交对象C6，master指针指向C6提交对象的索引

之后则可以删除iss53分支

Git分支的优势

以下这段引用说明了在Git中提倡使用分支的原因：

这和大多数版本控制系统形成了鲜明对比，它们管理分支大多采取备份所有项目文件到特定目录的方式。
所以根据项目文件数量和大小不同，可能花费的时间也会有相当大的差别，快则几秒，慢则数分钟。而Git的实现与项目复杂度无关，它永远可以在几毫秒的时间内完成分支的创建和切换。

（例如svn的分支是以直接copy来进行实现的，在merge的时候也是对两个文件夹进行操作。但是这样做的缺点就是代码管理中多了一个版本库代码的copy，而git只是在同一个文件夹中就可以完成一个或者多个branch的管理）

另外因为每次的提交对象时都记录了parent的提交对象，将来要合并分支时，寻找两个分支的共同祖先的工作基础已经自然而然地摆在那里了，所以实现起来非常容易
